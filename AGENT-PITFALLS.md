# Agent 踩坑记录

> 用途：每次启动子 Agent 前注入，避免重复犯同样的错误。

---

## 坑 1：Agent 无法写文件

**现象**：子 Agent（Plan Agent）可能处于 read-only 模式，无法调用 Write/Edit 工具写文件。

**解决**：子 Agent 只需输出完整的 Markdown 内容，由主 Agent（Claude）负责写文件。

**子 Agent prompt 中必须加**：
```
如果你无法写文件，直接输出完整 Markdown 内容即可，由主 Agent 写入。
```

## 坑 2：文档写成"方案比选报告"而非"确定设计"

**现象**：SAD 中每个技术选型都列了 3 个方案 + 优缺点对比 + "建议"选项，像咨询报告不像架构设计。

**解决**：
- 技术选型直接拍板一个方案，说明理由，不列备选
- 禁止出现"建议""候选""推荐""可以考虑"等词
- 用"确定选型""最终选择"替代"建议方案"

**子 Agent prompt 中必须加**：
```
这是确定性文档，不是方案比选报告。每个技术决策直接给出最终选择+理由，不要列多个选项对比。禁止使用"建议""候选""推荐"等词汇。
```

## 坑 3：文档末尾带"待确认事项"

**现象**：SRS/SAD 最后都有一个"待确认事项"表格，破坏文档的确定性。

**解决**：文档必须是"已确认的最终版"，所有决策都在正文中。如需确认项，单独输出到审核清单。

**子 Agent prompt 中必须加**：
```
文档末尾不得出现"待确认""待决事项""请确认"等章节。文档本身必须是确定的、自包含的。
```

## 坑 4：功能需求描述笼统

**现象**：需求写成"系统支持账号管理"一句话，没有前置条件、操作流程、异常处理。

**解决**：每个功能需求至少包含：描述、前置条件（如需要）、验收标准。

## 坑 5：数据模型缺少约束

**现象**：数据库表只有字段名和类型，缺少 NOT NULL、UNIQUE、DEFAULT 等约束。

**解决**：每个字段必须标注约束条件。

## 坑 6：非功能需求无法量化

**现象**："系统应该快速响应"——没有数字。

**解决**：所有非功能需求必须有数字指标（秒、MB、百分比、次数等）。

## 坑 7：前后矛盾

**现象**：前面说"不做 Docker"，后面非功能需求写"Docker Compose 一键部署"。

**解决**：写完后自查所有约束是否一致。

## 坑 8：RAG 流程遗漏关键环节

**现象**：RAG 问答流程只写了检索和生成，遗漏了 Embedding 步骤、相似度阈值判断、降级策略。

**解决**：RAG 流程必须完整覆盖：预处理 → Embedding → 检索 → 判阈 → 生成/降级。

## 坑 9：缺少初始化数据需求

**现象**：系统需要演示，但文档没提预置数据。

**解决**：需求阶段应说明是否需要预置示例 FAQ，实现阶段应准备初始化脚本。
