# Agent 迭代工作流

> 用途：说明主 Agent（Claude）与子 Agent（Plan Agent）之间的协作方式。

---

## 整体流程

```
┌─────────────────────────────────────────────────────┐
│ 用户审核 ←→ Claude（主 Agent）←→ Plan Agent（子 Agent）│
│   ↑            ↑                      ↑             │
│   │            │                      │             │
│   │      最终决策者              产出文档             │
│   │      审核子Agent产出         写入 docs/ 目录       │
│   │      多轮迭代直到达标                              │
└─────────────────────────────────────────────────────┘
```

## 角色分工

| 角色 | 职责 |
|------|------|
| **Claude（主 Agent）** | 接收用户指令 → 理解阶段目标 → 派发子 Agent 产出 → **审核产出质量** → 指出问题 → 让子 Agent 修正 → 多轮迭代 → 达标后交给用户审核 |
| **Plan Agent（子 Agent）** | 接收任务描述 + PROJECT-BRIEF.md → 产出文档写入 `docs/` → 等待 Claude 审核反馈 → 修正后重新交付 |
| **用户** | 每个阶段最终审核，确认后进入下一阶段 |

## 迭代规则

1. Claude 给子 Agent 的 prompt 必须包含：阶段目标、输入文档路径、输出文件路径、格式要求
2. 子 Agent 产出后，Claude **必须**逐项审核，列出通过项和问题项
3. 如果发现问题，Claude 给出具体修改指令，让子 Agent 修正，**不允许 Claude 直接修改文档**
4. 迭代直到 Claude 判定达标，才交给用户审核
5. 用户审核通过后，锁定该阶段产出，进入下一阶段

## 阶段划分

> 按软件工程标准流程，逐阶段推进：

| 阶段 | 产出文档 | 说明 |
|------|----------|------|
| **1. 需求分析** | `docs/SRS.md` | 需求规格说明书（IEEE 830 风格） |
| **2. 架构设计** | `docs/SAD.md` | 系统架构设计说明书 |
| **3. 详细设计** | `docs/DD.md` | 数据库 DDL + API 接口定义 + 前端路由设计 |
| **4. 编码实现** | `backend/` `frontend/` | 分模块编码，每模块完成后测试 |
| **5. 测试** | `docs/TEST.md` | 测试用例 + 测试报告 |
| **6. 交付** | 最终打包 | 源码 + 文档 + 启动脚本 |

## 文档质量标准

一份合格的阶段文档必须满足：

1. **确定性**：不含"建议""待确认""候选方案""可能"等不确定词
2. **完整性**：覆盖 PROJECT-BRIEF.md 中所有需求点
3. **可追溯**：需求编号贯穿始终，验收标准关联到需求
4. **可操作**：后续阶段可以直接基于此文档工作，不需要再猜测
5. **自包含**：不需要额外口头解释就能读懂

## 子 Agent 启动模板

每次启动子 Agent 时，Claude 应按以下格式构造 prompt：

```
你是一位资深软件工程架构师。请基于以下项目背景撰写 [阶段名称] 文档。

## 项目背景
[注入 PROJECT-BRIEF.md 核心内容]

## 任务要求
- 输出文件：D:\code\demo\enterprise-knowledge-rag\docs\[文件名]
- 格式要求：[具体格式要求]
- 约束：[本阶段须遵守的约束]
- 禁止事项：[不可出现的内容]

## 上一阶段产出（如有）
[引用上一阶段的文档内容]

请直接产出完整文档，不要中途暂停，不要输出"待确认"。
```
