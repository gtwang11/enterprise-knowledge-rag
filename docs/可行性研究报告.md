| 版本 | 日期 | 说明 | 作者 |

## 1. 引言

### 1.1 课题来源

课题六《运维数字员工的建设》，选自《广东公司提供实训课题2026年版》。

### 1.2 要干什么

做一个运维数字员工门户——用本地大模型 + RAG 私有知识库搭一个 Web 系统。一线运维人员可以在上面自助查询技术问题的答案，AI 答不了的自动生成工单流转给资深专家，专家处理完以后把方案一键录入知识库，下次 AI 就能回答了。

### 1.3 术语

| 简称 | 全称 |
|------|------|
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| LLM | Large Language Model，大语言模型 |
| FAQ | Frequently Asked Questions，常见问题解答 |
| Embedding | 文本向量化 |

## 2. 技术可行性

### 2.1 要用什么技术

| 层次 | 选型 | 原因 |
|------|------|------|
| 后端 | Python FastAPI | 异步框架，自带接口文档，跟 ML 生态兼容好 |
| 前端 | Vue 3 + Element Plus | 成熟的企业级 UI 组件库，中文社区活跃 |
| 数据库 | SQLite | 单机够用，不需要单独起数据库服务 |
| 向量库 | ChromaDB | Python 原生，嵌入式运行，轻量 |
| 大模型 | Ollama + DeepSeek-7B | 本地推理，不依赖云端，量化后 4.5GB |
| 向量模型 | Ollama + nomic-embed-text | 768 维，274MB，中文能用 |

### 2.2 关键问题能不能解决

**RAG 管线能不能跑通？**

文本预处理（jieba 分词 + 去停用词）→ Ollama embedding 接口转成向量 → ChromaDB 存向量 → 查询时余弦相似度检索 → LLM 用检索到的上下文生成回答。这个链路技术都是现成的，组合起来就行。

Ollama 的 `/api/embeddings` 和 `/api/generate` 接口文档清楚，调起来跟普通的 REST API 一样，不复杂。

**大模型在单机上能跑吗？**

DeepSeek-LLM-7B 4-bit 量化版内存占用约 4.5GB。加上 embedding 模型 ~0.3GB，后端 ~0.3GB，ChromaDB 和 SQLite 可以忽略不计，系统空闲内存约 2.5GB——在 8GB 内存的机器上跑没问题。

**FAQ 批量导入效率？**

1025 条 FAQ 逐条调 Ollama embedding 接口大概要 8-10 分钟。做了分批处理（每 10 条休息 3 秒）防止 Ollama 过载后拒绝响应。

### 2.3 结论

技术上没卡点，选的全是成熟的开源组件，文档和社区资源都够。可行。

## 3. 经济可行性

### 3.1 成本

- 硬件：一台 4C8G 100GB 硬盘的 PC，成本约 3000-5000 元
- 软件：全开源，零授权费
- 开发人力：1 人天
- 运维：部署完基本不用管

### 3.2 效益

没法算具体 ROI，但定性地看：
- 一线人员查问题不用每次都打电话找人，省时间
- 知识库越用越丰富，减少对老员工的依赖
- 工单流转自动化，相比微信群/邮件沟通效率高
- 系统把处理过程和结果都记下来了，方便后面追溯

对一个几十人的运维团队来说，这个系统肯定比没有强。

### 3.3 结论

成本极低（硬件现成的），收益实实在在。可行。

## 4. 操作可行性

### 4.1 用户能不能用起来

三类用户：
- 一线运维：就一个搜索框打字问问题，跟用百度差不多
- 资深专家：接单 → 写方案 → 点确认，流程清晰
- 管理员：表单填数据，跟大多数后台系统一样

学习成本很低，应该不需要培训。

### 4.2 部署和维护

部署就三步：装 Python → pip 装依赖 → 跑 `python main.py`。写了一个环境配置清单跟着做就行。

日常维护就是偶尔看下日志，数据库和 ChromaDB 都是文件存储，备份直接拷文件夹就行。

### 4.3 结论

使用门槛够低，运维成本够小。可行。

## 5. 风险与应对

| 风险 | 概率 | 影响 | 怎么办 |
|------|------|------|--------|
| Ollama 模型下载慢或失败 | 中 | 高 | 首次启动联网下载，后续离线可用；准备模型备份 |
| ChromaDB 版本升级不兼容 | 中 | 中 | requirements.txt 锁版本范围，不盲目追新 |
| LLM 生成质量不稳定 | 低 | 中 | 设相似度阈值 0.65 兜底，检索结果不够就不要让 LLM 硬编 |
| 单机性能不够 | 低 | 中 | 选 4-bit 量化模型，FAQ 量控制在万级别以内 |
| 知识库内容质量问题 | 中 | 低 | 管理员可以编辑和下线 FAQ |

---

## 6. 总体结论

技术选型成熟靠谱，成本几乎为零，使用门槛低，风险可控。项目可行，建议开搞。
