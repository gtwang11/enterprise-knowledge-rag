| 版本 | 日期 | 说明 | 作者 |

| V1.0 | 2026-06-06 | 初稿 | 架构组 |

---

## 1. 概述

数据库用 SQLite，文件位置 `backend/data/eknowledge.db`。WAL 模式开启，busy_timeout 设了 5 秒，支持并发读。

ORM 用 SQLAlchemy 2.0，所有表通过 `Base.metadata.create_all()` 自动建。

---

## 2. ER 图（文字版）

```
users ──1:N── password_history    (密码变更记录)
users ──1:N── token_blacklist     (登出后的黑名单 token)
users ──1:N── operation_logs      (管理员操作日志)
users ──1:N── qa_history          (问答历史)
users ──1:N── tickets (创建者)     (工单)
users ──1:N── tickets (处理者)     (工单)
users ──1:N── faq (创建者/修改者)   (FAQ)

tickets ──1:N── ticket_logs       (工单流转日志)
tickets ──1:1── faq               (source_ticket_id，方案入库来源)

faq ── 向量同步 ── ChromaDB faq_vectors
```

## 4. ChromaDB 向量存储

### 4.1 集合设计

集合名：`faq_vectors`（活跃），重建时临时集合叫 `faq_vectors_v2_{uuid8}`。

每条向量记录：
```
id:       str(faq.id)
embedding: 768 维 float 列表
metadata: {
    faq_id: faq.id,
    question: faq.question[:500],
    category: faq.category
}
document: faq.question
```

空间距离函数：cosine。

### 4.2 向量化文本拼接

调 embedding 时用的文本是这样拼的：

```
f"{question} {category} {keywords} {answer[:200]}"
```

取前 200 字答案是因为 embedding 模型对长文本效果会下降，而且 FAQ 的答案大多开头就是结论。

## 5. 索引策略

SQLite 表上的索引：

| 表 | 索引字段 | 原因 |
|----|---------|------|
| users | username (unique) | 登录时按账号名查 |
| users | role, status | 账号列表筛选 |
| faq | category | FAQ 分类筛选 |
| faq | status | 检索时只查 published |
| faq | source_ticket_id | 工单关联 FAQ |
| tickets | ticket_no (unique) | 按编号查 |
| tickets | status | 工单列表筛选 |
| tickets | creator_id, handler_id | 按人查工单 |

ChromaDB 自带 HNSW 索引，不需要手动建。
