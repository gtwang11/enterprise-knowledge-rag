## 1. 测试是干什么的

就是把系统从头到尾跑一遍，看看有没有 bug。我们测试了全部 36 个接口，覆盖了登录、问答、工单、FAQ 管理、账号管理等模块。测试分两轮：第一轮找出 15 个问题，修完之后第二轮全过了。

测试环境就是开发机：Windows 11，4 核 8G 内存，Python 3.12，Chrome 浏览器。Ollama 在本地 11434 端口跑两个模型：deepseek-llm-7b-chat:q4（做推理）和 nomic-embed-text（做向量化）。

测试方法比较简单粗暴：用 curl 和 Python requests 调接口，看返回码和返回内容对不对。边界的场景也测了——空输入、超长文本、乱码、特殊字符。安全方面测了越权、SQL 注入、XSS，没发现问题。

## 4. 安全测试

都是手工测的：

- JWT 用了 HS256，密钥写在 config.py 里（不是硬编码到各处）
- 密码 bcrypt 哈希，cost=12
- SQL 注入：用 SQLAlchemy 的参数化查询，没测出来注入点
- XSS：Vue 默认把 {{ }} 里的东西都转义了，没找到注入点
- 没登录直接调 API → 全部 401
- operator 调 admin 的接口 → 全部 403
- 改密码时会检查历史，前 3 次用过的不能再用
- 登出后拿旧 token 继续用 → 401
- 连错 5 次密码锁 15 分钟
- CORS 只开了 localhost，没有 *

## 5. 发现的 bug

第一轮测出来 15 个问题：

| 等级 | 个数 | 都修了吗 |
|------|------|---------|
| 要命的 (P0) | 5 | 全修了 |
| 中等 (P1) | 5 | 全修了 |
| 小问题 (P2) | 5 | 全修了 |

具体是哪些：

- **P0**：密码写死 "123456"、ticket 表少了个 assigned_at 列、JWT 返回的过期时间写死 30 分钟但实际是 8 小时、ChromaDB 版本不兼容、embedding 零向量没检测
- **P1**：向量库 1025 条只向量化了 298 条、reindex 没回滚、LLM 报错被吃了、限流计数器只增不减内存泄漏、前端路由没鉴权
- **P2**：批量导入 Ollama 超时、特殊字符输入被洗成空串、CORS 配了 * + credentials、create_user 有个没用的参数、FAQ 导入时空分类和错分类行为不一样

第二轮全回归了一遍，都过了，没引入新 bug。

## 6. 结论

36 个接口，81 条用例，第一轮发现 15 个 bug 全修了，第二轮回归 0 新增缺陷。性能和安全都还行。功能上该有的都有了，可以交。
