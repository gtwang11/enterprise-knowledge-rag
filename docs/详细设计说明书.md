## 1. 后端模块详细设计

### 1.1 RAG 管线 (rag_pipeline.py)

这是系统最核心的类，把整个问答流程串起来。

**类：RAGPipeline**

方法 `execute(question: str) -> dict`：

```
1. TextPreprocessor.process(question)
   输入：用户原始问题
   输出：清洗+分词后的文本
   异常：全特殊字符 → 返回原文 + 日志告警

2. EmbeddingManager.embed(cleaned)
   输入：预处理后的文本
   输出：768 维 float 列表
   异常：Ollama 不可用 → 向上抛异常，上层返回降级消息

3. VectorStore.search(query_vector, k=5)
   输入：768 维向量
   输出：[{faq_id, question, similarity}, ...] 最多 5 条
   内部：cosine distance → similarity = 1 - distance

4. 阈值判断
   similarity >= 0.65 → 走 LLM 生成
   similarity < 0.65  → 返回 "无法找到相关答案，是否创建工单？"

5. LLMManager.generate(prompt)
   输入：拼好的 prompt（系统指令 + 检索到的 FAQ 上下文 + 用户问题）
   输出：LLM 生成的回答文本
   异常：LLMError → 返回 fallback 消息 + 引用
```

### 1.2 文本预处理 (text_preprocessor.py)

```
正则清洗: re.sub(r'[^一-龥a-zA-Z0-9\s]', ' ', text)  # 去特殊字符
多余空白: re.sub(r'\s+', ' ', cleaned).strip()

如果清洗后为空:
  → 返回原始 text.strip()
  → 写 WARNING 日志

jieba 分词:
  jieba.cut(cleaned) → 去停用词 → 空格连接
  分词后为空 → 回退到 cleaned
  jieba 没装 → 返回 cleaned
```

### 1.3 Embedding 管理 (embedding_manager.py)

```
POST http://localhost:11434/api/embeddings
  body: {"model": "nomic-embed-text", "prompt": text}
  timeout: 30s
  proxies: {"http": None, "https": None}  # 不走代理

返回处理:
  取 resp.json()["embedding"]
  检查长度 == 768，短了补零，长了截断
  注意：不再吞异常，失败了直接抛给上层
```

### 1.4 向量存储 (vector_store.py)

**活跃集合热切换机制**：

模块级变量 `_active_collection_name` 指向当前活跃的 ChromaDB 集合名。平常就是 `faq_vectors`。重建索引时后台写一个新集合（比如 `faq_vectors_v2_bd5ea902`），写完以后原子改指针指向新集合，再删旧的。

```
日常操作:
  add_faq()     → 写当前活跃集合
  delete_faq()  → 从活跃集合删
  search()      → 从活跃集合查

重建索引:
  rebuild_index_background(faqs) → 后台线程:
    1. 创临时集合 faq_vectors_v2_{uuid8}
    2. 逐条 embedding + 写入
    3. _active_collection_name = 临时集合名
    4. 删旧集合
  返回: {"thread_id": "xxx", "total": 1025, ...}
```

progress 状态存在模块级 dict `_reindex_status`，`/api/faq/reindex/status` 接口直接读它。

**search 方法细节**：

```
collection.query(
    query_embeddings=[vector],
    n_results=k,
    include=["metadatas", "distances", "documents"]
)
→ 遍历结果，distance 转 similarity (1 - distance)
→ 返回 [{faq_id, question, similarity}]
```

**重试机制**：

所有 ChromaDB 操作包在 `_retry()` 里，失败自动重试 3 次，间隔 5 秒。

### 1.5 LLM 管理 (llm_manager.py)

**LLMError 异常类**：继承 Exception，带一个 `fallback` 属性存放降级消息。

```
generate(prompt) → str:
  POST http://localhost:11434/api/generate
    model: "qwen2.5:7b"
    prompt: prompt, stream: False
    options: {temperature: 0.1, num_predict: 1024}
    timeout: 30s

异常处理（不再吞）:
  requests.Timeout        → LLMError("超时", fallback="...")
  requests.ConnectionError → LLMError("连不上", fallback="...")
  requests.HTTPError       → LLMError("服务端报错", fallback="...")
  其他                     → LLMError("未知错误") + ERROR 日志

正常但空响应 → LLMError("空响应")
```

上层 rag_pipeline 捕获 LLMError，用 `e.fallback` 作为答案，`str(e)` 写入 message 字段。

### 1.6 工单状态机 (ticket_service.py)

状态转移图：

```
                    ┌──────────┐
        operator创建 → pending ─────┐
                    └────┬─────┘    │
                         │          │ expert 驳回
               expert 领取│          ▼
                         │     rejected
                         │
                         ▼
                    ┌──────────┐
                    │processing│
                    └────┬─────┘
                         │
               expert 提方案│
                         ▼
                    ┌──────────┐
                    │ resolved │
                    └────┬─────┘
                         │
          operator 确认 ──┼── operator 不确认
                    ┌────▼───┐  ┌──────────┐
                    │confirmed│  │ reopened │──→ expert 重新处理
                    └─────────┘  └──────────┘
                         │
              expert 一键入库│
                         ▼
                   FAQ 表 + 向量库
```

每次状态变更都记到 `ticket_logs` 表里。

关键校验：
- `claim`：只有 expert 能操作，工单状态必须是 pending
- `submit_solution`：只有当前 handler 能操作，状态必须是 processing
- `reject`：只有 expert 能操作，必须填驳回原因
- `confirm/unconfirm`：只有工单创建者能操作，状态必须是 resolved
- `publish_faq`：只有 handler 能操作，状态必须是 confirmed

### 1.7 前端路由守卫 (router/index.ts)

```typescript
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  const role = localStorage.getItem('role')

  // 目标需要登录，你没 token → 去登录页
  if (to.meta.requiresAuth !== false && !token) {
    return next({ name: 'Login', query: { redirect: to.fullPath } })
  }

  // 目标限制了角色，你角色不对 → 403
  const roles = to.meta.roles as string[] | undefined
  if (roles?.length && role && !roles.includes(role)) {
    return next({ name: 'Forbidden' })
  }

  next()
})
```

### 1.8 限流中间件 (middleware/rate_limit.py)

只对 `POST /api/auth/login` 生效。

```
同一个 IP，60 秒窗口内最多 10 次。
数据存在内存 defaultdict 里。

每次请求:
  过滤掉窗口外的旧时间戳
  如果列表为空 → 删 key（防止内存泄漏）
  如果列表长度 >= 10 → 429
  否则追加当前时间戳

每 5 分钟全量扫描一次，清理不活跃的 IP。
```

## 2. 关键算法

### 2.1 余弦相似度

ChromaDB 返回的是 cosine distance，转成 similarity：

```
similarity = 1.0 - distance
distance 范围 [0, 2], similarity 范围 [-1, 1]
threshold = 0.65（对应 distance ≤ 0.35）
```

### 2.2 指数退避重试

后台向量化失败重试：

```
第 1 次失败 → 等 2^1 = 2 秒
第 2 次失败 → 等 2^2 = 4 秒
第 3 次失败 → 放弃，记 ERROR 日志
最多重试 3 次。
```

## 3. 接口约定

所有 API 返回统一格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "timestamp": 1780754676000
}
```

错误时 code 不是 200，message 里说明原因，data 为 null。

分页返回：

```json
{
  "code": 200,
  "data": {
    "items": [...],
    "total": 1025,
    "page": 1,
    "page_size": 20,
    "total_pages": 52
  }
}
```
