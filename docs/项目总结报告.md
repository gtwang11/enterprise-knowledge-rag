## 1. 项目概述

### 1.1 课题来源

课题六《运维数字员工的建设》，选自《广东公司提供实训课题2026年版》。

### 1.2 项目背景

企业内部 IT 运维部门的一线运维人员（基层工程师）在日常工作中频繁遇到各类技术问题。传统模式下，问题需要逐级上报或电话咨询资深专家，效率低且重复性高。本系统通过部署本地大模型和 RAG 私有知识库，构建运维数字员工门户，使一线运维人员能够自助查询问题答案，形成"自助查询 → AI 兜底 → 工单流转 → 人工处理 → 回访确认 → 知识入库"的完整闭环。

## 3. 技术方案

### 3.1 总体架构

```
Vue 3 前端 ←→ FastAPI 后端 ←→ SQLite + ChromaDB
                        ←→ Ollama (deepseek-llm-7b-chat + nomic-embed-text)
```

### 3.2 技术栈

| 层次 | 技术选型 | 理由 |
|------|---------|------|
| 后端框架 | FastAPI | 异步高性能，自动 OpenAPI 文档 |
| 前端框架 | Vue 3 + Element Plus | Composition API，成熟中文 UI 库 |
| 数据库 | SQLite | 零配置零进程，适合单机 |
| ORM | SQLAlchemy 2.0 | 类型安全，参数化防注入 |
| 向量库 | ChromaDB | 轻量 Python 原生，嵌入式运行 |
| LLM | Ollama + deepseek-llm-7b-chat | 本地推理，7B 参数量化 4.5GB |
| Embedding | Ollama + nomic-embed-text | 768 维，274MB，支持中文 |
| 分词 | jieba | 中文分词，去停用词 |
| 鉴权 | JWT (HS256) | 对称签名，适合单机 |
| 密码 | bcrypt (cost=12) | 业界标准 |

### 3.3 核心创新点

1. **RAG 热切换重建**：向量索引全量重建采用"写入临时集合 → 原子切换"方案，实现零停机更新，重建期间查询不受影响。
2. **LLMError 异常体系**：按错误类型（Timeout/Connection/HTTPError）区分，分别返回针对性降级消息，避免异常被吞没。
3. **批量向量化重试**：后台向量化任务支持 3 次指数退避重试，防止 Ollama 短暂不可用导致向量化失败。
4. **限流定期清理**：内存限流计数器每 5 分钟清理不活跃 IP，防止内存泄漏。
5. **知识闭环**：工单方案一键录入知识库，形成"自助查询→AI兜底→工单流转→人工处理→知识入库"的完整闭环。

## 6. 经验与反思

### 6.1 技术经验

1. **ChromaDB 版本管理**：开源向量库版本迭代快，0.x 和 1.x API 差异大。应在 requirements.txt 中明确版本约束（`>=0.5.0,<2.0.0`），避免自动升级导致的兼容性问题。

2. **Embedding 质量检查**：不能仅检查向量维度，还需检查全零向量。零向量在余弦距离计算中会导致所有文档相似度为 1.0，返回错误的检索结果。

3. **LLM 异常处理**：不应捕获所有异常后返回统一降级字符串，而应区分超时、连接失败、服务端错误等不同情况，分别返回有针对性的降级消息。

4. **阻塞式索引重建不可取**：全量重建向量索引如果采用同步阻塞方式，会导致数分钟的查询中断。采用"临时集合 + 原子切换"的方案可以实现零停机。

### 6.2 项目管理的反思

1. **配置与代码分离**：密码、密钥、URL 等敏感信息应与代码分离，通过环境变量注入。
2. **前后端联调**：开发过程中前后端应保持接口文档同步（FastAPI 自动生成的 OpenAPI 文档在此发挥重要作用）。
3. **分批测试**：批量导入 1000+ 条数据时，需要考虑 Ollama 的并发处理能力，添加批次间延迟。

### 6.3 后续改进建议

| 优先级 | 建议 | 说明 |
|--------|------|------|
| P1 | 前端 E2E 测试 | 使用 Playwright/Cypress 覆盖核心用户旅程 |
| P1 | embedding 缓存 | 对已向量化的 FAQ 内容做 hash 缓存，避免重复调用 Ollama |
| P2 | 日志轮转 | 当前日志文件持续增长，建议引入 RotatingFileHandler |
| P2 | 配置热更新 | 部分配置（如阈值）支持通过管理后台动态调整 |
| P3 | 前端 i18n | 当前仅中文，预留国际化能力 |

## 7. 展望

### 7.1 扩展方向

1. **多模态知识库**：支持图片、截图形式的 FAQ（需要多模态 Embedding 模型）
2. **相似问题推荐**：用户输入问题时，推荐类似的历史问答
3. **知识库质量评估**：统计 FAQ 被命中频率，标记低质量内容供管理员优化
4. **移动端适配**：响应式布局，适配手机浏览器

### 7.2 结语

运维数字员工门户系统实现了基于本地大模型和 RAG 私有知识库的智能运维问答平台，完成了从需求分析、架构设计、编码实现到测试交付的完整软件工程流程。系统采用轻量级技术栈，可在 4C8G 100GB 的单台 PC 上完整运行，满足企业内部 IT 运维的知识管理需求。
