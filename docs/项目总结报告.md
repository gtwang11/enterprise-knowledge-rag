## 1. 项目是干嘛的

选题是"运维数字员工的建设"，来源是广东公司实训课题六。简单说就是做一个企业内部 IT 运维的智能问答系统。

场景是这样的：一线运维人员（基层工程师）平时遇到各种技术问题，传统方法是打电话问老员工或者逐级上报，效率低、重复性高。我们的想法是用本地大模型 + 知识库做一个自查询的 Web 系统——问问题，AI 从知识库里找答案；找不到的就建工单，让资深专家来处理，处理完再把方案加回知识库。这样下次同样的问题 AI 就能回答了，不用每次都找老员工。

整个过程就一个闭环：**自助查询 → AI 兜底 → 工单流转 → 人工处理 → 回访确认 → 知识入库 → 自助查询**。

## 2. 做完了什么

### 2.1 功能

七个模块全部做完了：

| 模块 | 做了什么 |
|------|---------|
| 用户认证 | 登录/登出、JWT 鉴权、密码修改、输错 5 次锁号、IP 限流 |
| 账号管理 | 创建/编辑/删除/冻结/解冻/重置密码，带角色权限控制 |
| FAQ 知识库 | 增删改查、Excel/CSV/JSON 批量导入、向量同步、全量重建索引 |
| RAG 问答 | 中文分词 → 768 维向量化 → ChromaDB 检索 → LLM 生成，带阈值判断和降级 |
| 工单管理 | 创建/领取/提方案/驳回/回访确认/一键入库，完整状态流转 |
| 仪表盘 | 三个角色不同的统计看板 |
| 系统基础 | 种子数据、CORS、异常处理、操作日志 |

一共 36 个 API 接口，9 张数据库表，前后端加起来大概 3200 行代码。

### 2.2 满足的约束

课题要求的那几条都做到了：
- 大模型本地跑，没调云端 API（Ollama + DeepSeek-7B）
- 只做 Web，没做 APP
- 没用 Docker，直接命令行启动
- 4C8G 100GB 单机能跑
- 全中文

## 3. 技术方案

整体是 Vue 3 前端 + FastAPI 后端 + SQLite + ChromaDB，外面接一个 Ollama。

选型当时考虑了几点：
- FastAPI 够轻、写异步方便、自带接口文档
- Vue 3 + Element Plus 做后台管理很顺手
- SQLite 单机够用，不用单独起数据库进程
- ChromaDB 也是嵌入式跑，没有独立服务
- deepseek-llm-7b-chat 量化后 4.5GB，在 8G 机器上能跑
- nomic-embed-text 做向量化，274MB，768 维，中文能用

核心流程：
```
用户输入问题 → jieba 分词 → Ollama 转成 768 维向量
→ ChromaDB 搜 Top-5 最相似的 FAQ → 相似度 ≥ 0.65 就丢给 LLM 生成回答
→ 不够 0.65 就提示"找不到，要建工单吗"
```

## 4. 开发过程

整个项目在一天内做完，分成了几个阶段：

1. 先搭架子：FastAPI 启动、Vue 项目创建、SQLite 建表
2. 写核心链路：RAG 管线（预处理 → embedding → 检索 → LLM）
3. 补业务模块：工单状态流转、FAQ 管理、账号管理
4. 联调：前后端对接，调 CORS、token 刷新
5. 测试：接口全跑一遍，修了 15 个 bug
6. 写文档

### 踩过的坑

说几个印象比较深的：

**ChromaDB 版本冲突**。requirements.txt 里写了 `chromadb>=0.5.0`，结果 pip 装了 1.5.9。0.x 到 1.x API 变了不少，比如 `delete(where={})` 在 1.x 里不行，得先 `get(ids)` 再 `delete(ids)`。后来把版本锁成 `>=0.5.0,<2.0.0`，加了线程锁保护客户端创建。

**Embedding 零向量**。Ollama 有时候不报错但返回一串 0，没检查就存进去了。零向量跟谁算余弦相似度都是 1.0，导致返回完全不搭边的结果。修法是 embedding 完检查一下是不是全零。

**LLM 报错被吃掉**。原来 `llm_manager.py` 里 try-except 捕获了所有异常，返回一句"AI 服务暂不可用"。问题是超时、连不上、模型报错全都返回同一句话，外面没法区分。后来改成自定义 LLMError 异常，Timeout / Connection / HTTP 分开处理。

**同步 reindex 阻塞查询**。最开始重建索引是同步的——先清空再逐条 rebuild，1025 条跑完要七八分钟，这期间所有查询全挂。后来改成了原子热切换：后台往一个临时集合里写，写完了把集合指针一切，旧的删掉，查询完全不受影响。

**限流计数器内存泄漏**。`rate_limit.py` 里有个 dict 存每个 IP 的请求时间，只增不减。加了定期清理逻辑，空列表的 key 直接删掉。

## 6. 测试结果

功能测试：81 个用例，第一轮 15 个 bug，全修了，第二轮 0 新增。

性能：登录 <200ms，查列表 <100ms，问答含 LLM 大概 3-10 秒。向量检索 <50ms。1025 条重建索引大概跑 8 分钟（后台异步）。

安全：JWT + bcrypt + SQLAlchemy 参数化防注入 + Vue 默认转义防 XSS + 角色鉴权 + 登录锁定 + 限流。没测出明显的洞。

## 7. 不足和后续

做得不够的地方：
- 没有前后端自动化测试（现在全靠手工调接口）
- 日志没有轮转，时间长了文件会很大
- embedding 没有缓存，重建索引时重复的文本每次都重新调 Ollama
- 前端没有做响应式，手机上没法用
- 配置项改了需要重启服务才能生效

如果接着做，想加的功能：
- 相似问题推荐（输问题的时候自动提示）
- FAQ 质量统计（哪些被命中的多，哪些一直没人用）
- 支持图片 FAQ（比如配截图说明的，不过那就需要多模态模型了）
